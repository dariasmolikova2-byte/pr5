Завдання 5. Використовуючи інструменти штучного інтелекту, необхідно здійснити опис кожного кроку (ітерації), виконати її аналітичне опрацювання та включити результати до звіту. 

Ітерація 1: Ідентифікація Основних Сутностей-Довідників 

На першому етапі аналізу виділяємо основні незалежні сутності, які є базовими "довідниками" для системи розкладу. Вони містять статичну інформацію. 

Група: Описує студентські групи (ID Групи, номер, курс, спеціальність). 

Корпус: Описує навчальні будівлі (ID Будівлі, назва). 

Викладач: Описує персонал (ID Викладача, П.І.Б., посада). 

Кафедра: Описує структурні підрозділи (ID Департаменту, назва). 

Предмет: Описує дисципліни (ID Предмету, назва). 

Тип заняття: Описує формат уроку (лекція, практика) (ID Типу урока, назва). 

Пара / День тижня: Описують часові та календарні слоти розкладу (ID Слота, ID Дня). 

Ітерація 2: Аналіз Залежних Сутностей та Ієрархії 

На цьому етапі розглядаємо сутності, які залежать від вже ідентифікованих довідників через зовнішні ключі (Foreign Keys, FK). 

Аудиторія: Ця сутність залежить від Корпусу (ID Будівлі є FK). Кожна аудиторія (ID Кімнати) має бути прив'язана до конкретної будівлі. 

Викладач: Ця сутність залежить від Кафедри (ID Департаменту є FK). Кожен викладач належить до однієї кафедри. 

Ітерація 3: Центральна Асоціативна Сутність ("Таблиця Фактів") 

Центральним елементом, який зв'язує всю інформацію, є сутність Заняття (ID Класу). Вона виконує роль асоціативної таблиці або таблиці фактів розкладу, збираючи зовнішні ключі від більшості інших сутностей. 

Заняття включає такі зовнішні ключі: ID Групи, ID Викладача, ID Кімнати, ID Предмету, ID Типу предмета, ID Дня, ID Слота. 

Таким чином, один запис у таблиці "Заняття" унікально визначає, який предмет, яка група, який викладач, у якій аудиторії, у який день та у яку пару проводять який тип заняття. Це дозволяє однозначно побудувати повний розклад. 

Ітерація 4: Аналіз Обмежень Зв'язків та Композитних Ключів 

На останньому етапі розглядаємо додаткові обмеження, зазначені на схемі, які допомагають підтримувати цілісність даних та бізнес-правила. 

Нижній блок описує зв'язок між Викладачем та Типом заняття через композитний первинний ключ (PK): PK=(ID_Teacher,ID_LessonType). 

Це, ймовірно, є окрема таблиця-зв'язка, що фіксує, які саме типи занять (лекції, практики, лабораторії) дозволено або призначено вести конкретному викладачу. Це запобігає призначенню викладачу типу заняття, яке він не веде (наприклад, лаборант не може вести лекції). Обидва поля тут є зовнішніми ключами (FK). 

Результати Аналітичного Опрацювання 

Схема демонструє високий рівень нормалізації. Кожна сутність відповідає за зберігання інформації про окремий об'єкт предметної області, а зв'язки реалізовано через зовнішні ключі, що мінімізує надмірність даних. 

Ключове завдання системи: Ефективно управляти центральною сутністю Заняття, забезпечуючи, щоб комбінації семи зовнішніх ключів, які формують запис, не створювали конфліктів у розкладі (наприклад, щоб аудиторія чи викладач не були зайняті одночасно кількома заняттями). 

Висновок: 

Протягом виконання практичної роботи №5 було успішно здійснено ітеративне проектування бази даних для системи управління навчальним розкладом. На основі наданої предметної області розроблено вербальну модель та її логічне представлення у вигляді Entity-Relationship Diagram (ERD). 

Основні результати та досягнення: 

Створення нормалізованої структури: Модель даних повністю відповідає вимогам третьої нормальної форми (3NF), мінімізуючи надмірність і забезпечуючи цілісність даних. 

Централізація даних: Визначено центральну сутність «Заняття», яка через систему зовнішніх ключів (FK) об'єднує всі ключові довідники (Група, Викладач, Аудиторія, Предмет, День тижня, Пара). Це створює надійну основу для формування єдиного, несуперечливого розкладу. 

Фіксація бізнес-правил: Додано обмеження для зв'язку «Викладач – Тип заняття», що дозволяє системі контролювати розподіл навантаження та відповідність кваліфікації викладача до типу заняття. 

Таким чином, розроблена логічна модель є надійною, гнучкою та повністю готовою до наступного етапу — фізичної реалізації у відповідній реляційній системі керування базами даних (СКБД) з метою автоматизації процесу складання навчального розкладу.
